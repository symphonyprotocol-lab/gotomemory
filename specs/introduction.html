为什么是控制面，而不是又一个记忆库

不是 memory.add / search，而是决定它能否安全进入模型。

核心价值不在于再实现一个通用向量记忆库，而在于决定「哪些记忆、被哪个客户端、哪个模型、以什么权限、在什么任务中使用」——并让每一次暴露都可追踪、可撤销、可审计。

01

跨平台记忆护照

同一套个人、项目与团队记忆，可在 ChatGPT、Claude、Gemini、MCP 客户端和 API 消费方之间复用。

02

Policy-first 治理

读取、注入、导出是分开的权限。拥有搜索预览能力，并不等于能读取完整正文或注入上下文。

03

可见且可撤销

每次注入都返回 decision_id、记忆清单与省略原因。你可以随时暂停、撤销、删除或调整策略。

04

隐私分级默认安全

正文与完整摘要加密存储；private 默认确认后注入；secret 默认不自动注入、不生成 embedding。

05

适配器语义稳定

平台 API 变化时新增 adapter 版本，绝不静默改变记忆在目标模型中的权限与 payload 位置。

06

防 Prompt Injection

记忆以「用户授权上下文」注入，并明确声明它是上下文事实，不是高优先级系统指令。

三步走 · 写入 → 治理 → 注入

它是怎么工作的

每一步都映射到一个稳定的 Gateway API。最关键的 context/build 被设计为治理接口，而非普通检索接口。

1
POST /v1/memories

写入与分类

系统自动判定 type、sensitivity 与 freshness。检测到密钥、凭证等模式会自动上调敏感度——提交的敏感度是下限，不是上限。

2
POST /v1/context/build

治理式构建上下文

在解密正文之前完成策略过滤，按 token 预算确定性压缩，返回可注入上下文、decision_id、注入清单与 omitted 原因。

3
POST /v1/context/confirm

确认与注入

需要确认的记忆只回脱敏预览与一次性 token。确认后经 adapter 放入目标平台的低权限位置——不会被伪装成系统指令。

public < normal < private < secret

敏感度从第一天就内建在数据模型里。它同时决定：能否在搜索中预览、能否生成 embedding、以及默认的注入方式。

真实场景

它解决的，是日常里反复出现的问题

跨助手保留偏好

在 ChatGPT 里说「代码示例优先用 TypeScript」，之后在 Claude、Gemini 提问，系统自动注入这条偏好。

项目背景共享

在 Claude 讨论某仓库架构，系统提取项目记忆；换到 ChatGPT 继续时，目标、技术栈与历史决策都在。

敏感记忆保护

「我负责公司内部支付系统」这类背景，默认不会向第三方会话注入，除非你为当前项目明确授权。

团队协作

团队共享品牌语气、API 约定、部署流程。成员在不同 AI 平台工作时,都能得到一致的上下文。

一致的注入语义

记忆始终被放在目标平台的「低权限上下文」位置，权限低于系统/开发者规则。

快速开始

需要 Node 22.x 与 pnpm 11.x。启动 Gateway,然后用 CLI 创建、检索并构建上下文。

~/gotomemory — zsh
# 安装并构建
pnpm install && pnpm exec turbo run build

# 启动 Gateway（内存版 dev 后端）
PORT=8787 node apps/gateway/dist/index.js &

# 指向网关,创建一条偏好,然后检索 / 构建上下文
export GOTOMEMORY_URL=http://localhost:8787/v1 GOTOMEMORY_TOKEN=t1:u1
echo "prefers TypeScript" | node apps/cli/dist/bin.js memory create --type preference --json
node apps/cli/dist/bin.js memory search typescript --json
node apps/cli/dist/bin.js context build --task "write code" --json
